# Application Data Flow

This document explains how data moves through the AI Image Classifier, from
authentication and Google Drive ingestion to face-vector search and the final
API response. It describes the current implementation, not a proposed design.

## System boundaries

| System | Data it owns |
| --- | --- |
| Browser | Google access token, backend access/refresh tokens, and browser-only uploaded photos in `localStorage` |
| FastAPI | Request validation, authentication, orchestration, and response creation |
| Postgres | Users, Drive folder and image metadata, face metadata, ingestion state, search history, and result relationships |
| Google Drive | Original source images |
| Supabase Storage | Optional private copy of each ingested image's bytes |
| InsightFace | In-memory face detection and 512-value embedding generation |
| Qdrant | Face embeddings and the identifiers needed to join them back to Postgres |
| Celery/Redis | Production ingestion task delivery; it carries a job ID, not image bytes |

All protected operations are scoped to the authenticated `user_id`. Search
also applies the user filter inside Qdrant before limiting results, preventing
one user's faces from appearing in another user's result set.

## Important terminology

- **Drive image:** an image file discovered in a selected Google Drive folder.
- **Face:** one detected face inside an image. An image can have zero, one, or
  many faces.
- **Face embedding:** a 512-number vector produced by InsightFace for one face.
  Similar faces have vectors that are close under cosine similarity.
- **Image embedding:** the current application does **not** produce a general
  whole-image or text/semantic embedding. “Drive file embeddings” in this
  project means the face embeddings extracted from each Drive image.
- **Postgres/Qdrant join:** a `faces.id` UUID is also used as the Qdrant point
  ID and is stored in `faces.qdrant_point_id`.

## End-to-end overview

```text
Google sign-in
    -> Google token verified by backend
    -> user and Drive credentials saved in Postgres
    -> backend JWTs returned to browser

Selected Drive folder
    -> folder row and ingestion job created in Postgres
    -> worker receives only the ingestion job ID
    -> Drive image metadata listed
    -> each image downloaded into worker memory
    -> optional image copy uploaded to Supabase Storage
    -> InsightFace detects zero or more faces
    -> face metadata saved in Postgres
    -> face embeddings saved in Qdrant

User query image
    -> bytes uploaded to FastAPI and kept in request memory
    -> primary face embedding generated by InsightFace
    -> Qdrant searches only that user's stored face vectors
    -> Qdrant face IDs resolved to Postgres images/faces
    -> search history and ranked results saved in Postgres
    -> optional signed Supabase image URLs returned to browser
```

## 1. Authentication and user identity flow

### Google sign-in

1. The frontend requests the Google scopes `openid`, `email`, `profile`, and
   `drive.readonly`.
2. Google returns an access token to the browser.
3. The browser sends that token as `drive_access_token` to
   `POST /auth/google/session`.
4. The backend sends the token to Google's user-info endpoint. It trusts the
   verified response for the user's email, name, avatar, and Google subject ID.
5. Postgres creates or updates the `users` row and stores the Drive access token
   and optional refresh token.
6. The backend returns its own access and refresh JWTs. Their `sub` claim is the
   Postgres user UUID.
7. The frontend stores the Google and backend tokens in browser `localStorage`.

For every protected request, `get_current_user` verifies the backend access
JWT, rejects refresh tokens, parses the user UUID, and loads an active user from
Postgres. The resolved `user_id` is then used by all folder, image, job, and
search queries.

The email-only register/login routes exist for explicitly enabled development
environments. Production configuration is expected to disable them.

## 2. Google Drive folder registration flow

1. The browser calls `POST /ingestion/folders` with a Drive folder ID and an
   optional display name.
2. The backend authenticates the request and upserts a `user_folders` row using
   the pair `(user_id, drive_folder_id)`.
3. The folder begins in a pending state. At this point no Drive image bytes or
   embeddings have been processed.

Relevant persisted fields include the Drive folder ID, folder name, processing
status, total/processed/failed counters, timestamps, and the last error.

## 3. Ingestion job flow

The browser starts processing with
`POST /ingestion/folders/{folder_id}/start`.

1. The backend verifies that the folder belongs to the current user.
2. It creates an `ingestion_jobs` row with `status=queued`.
3. In production (`TASK_QUEUE_MODE=celery`), the API sends the job UUID to the
   Celery task named `ingestion.run_folder`. Redis transports the task message.
4. In local background mode, FastAPI starts the same job runner with a new
   database session.
5. The worker changes the job and folder states to running/processing.
6. It builds a Google Drive client using the credentials on the owning user's
   Postgres row.
7. It lists direct, non-trashed children of the folder, follows Drive pagination,
   and retains only files whose MIME type starts with `image/`.
8. The job and folder totals are updated to the discovered image count.
9. Each file is processed independently. Failed files receive one immediate
   retry; final failures are recorded without discarding successful files.
10. The job and folder are marked done or failed based on the stored counters.

Clients can poll `GET /ingestion/jobs/{job_id}` and
`GET /ingestion/folders/{folder_id}` for progress.

## 4. Per-Drive-image flow

For each Drive image, the worker performs the following sequence:

1. **Check for unchanged data.** If skipping is enabled and an already-complete
   image has the same Drive name, MIME type, and size, model processing is
   skipped and progress counters advance.
2. **Create relational identity.** The worker upserts an `images` row using
   `(user_id, drive_file_id)` and marks it processing. Creating the image UUID
   first gives later storage objects and face rows stable identifiers.
3. **Download bytes.** Google Drive returns the original bytes into worker
   memory. The configured ingestion-size limit is enforced.
4. **Optionally store the image.** If Supabase Storage is configured, the bytes
   are uploaded to a private bucket under:

   ```text
   {user_id}/{image_id}.{extension}
   ```

   Postgres stores the stable bucket and object key. It does not store a signed
   URL because signed URLs expire.
5. **Detect faces.** Pillow decodes the image as RGB; the array is converted to
   BGR for InsightFace. The shared CPU `buffalo_l` analyzer returns detections.
6. **Normalize detections.** Invalid detections are ignored. Valid detections
   contain a bounding box, confidence score, detector position index, and
   512-value embedding. Ingestion keeps every valid face; primary-face selection
   happens separately for a search query.
7. **Replace Postgres face rows.** Existing faces for a reprocessed image are
   deleted and new `faces` rows are inserted. Postgres stores bounding boxes,
   scores, image/user relationships, cluster relationship, and Qdrant point ID;
   it does not store the numeric embedding itself.
8. **Write vectors to Qdrant.** Each face embedding is stored as a point whose
   point ID is the Postgres face UUID. Its payload contains `user_id`, `face_id`,
   `image_id`, and optional `cluster_id`. The collection uses cosine distance.
9. **Commit completion.** The image is marked done with its face count, and job
   and folder counters advance in Postgres.

If no face is detected, the image is still a successfully ingested image with
`face_count=0`; it simply contributes no Qdrant vectors.

If processing fails, the current relational transaction is rolled back, the
image/file failure is recorded, and the folder runner can retry it once. Note
that Postgres and external services do not share one distributed transaction,
so operational cleanup may be needed if an external write succeeds immediately
before a later failure.

## 5. Face embedding representation

One Drive image can produce multiple records:

```text
images.id = I1
    faces.id = F1 -> Qdrant point F1 -> vector[512]
    faces.id = F2 -> Qdrant point F2 -> vector[512]
    faces.id = F3 -> Qdrant point F3 -> vector[512]
```

Postgres answers questions such as “which image contains this face?” and
“which user owns it?” Qdrant answers “which stored face vectors are closest to
this query vector?” Keeping the same face UUID in both systems connects those
responsibilities.

Qdrant payload indexes exist for `user_id`, `face_id`, `image_id`, and
`cluster_id`. The collection dimension is checked before writes/searches so a
different embedding model cannot silently write incompatible vectors.

## 6. User query image and search flow

The frontend submits a query image as multipart form data to
`POST /search?limit=N` with a backend bearer token.

1. FastAPI verifies that the upload has an image MIME type, is non-empty, and
   does not exceed `MAX_QUERY_IMAGE_BYTES`.
2. The bytes are read into request memory and passed to the synchronous search
   service through FastAPI's thread pool.
3. A `search_queries` row is created before face detection. This preserves a
   history entry even when no face is found.
4. InsightFace detects faces and selects the primary face. Its embedding is
   generated in memory.
5. If there is no primary face, the query is saved with
   `face_detected=false`, zero results, and measured latency. Qdrant is not
   called.
6. If a face is found, its embedding is sent to Qdrant. Qdrant performs cosine
   similarity search with a mandatory `user_id` filter and returns up to the
   requested limit.
7. Each Qdrant payload's `face_id` is parsed and resolved against Postgres.
   Missing or malformed/stale points are ignored.
8. Valid matches become ranked `search_results` rows containing the query,
   image, face, similarity score, and rank.
9. The `search_queries` row is updated with face-detection status, result count,
   top score, and latency.
10. The API reloads the query with related face/image metadata. If an image has
    a Supabase object key, it generates a temporary signed URL and returns it
    with the ranked result.

The uploaded query image itself is **not currently uploaded to Supabase**. The
`query_image_storage_key` field currently receives the upload filename, despite
its name. The raw query bytes and generated query embedding exist only during
the request and are then discarded. Search metadata and matched result rows
remain in Postgres.

`GET /search/{query_id}` can retrieve the persisted search and regenerate fresh
signed URLs, but only for the user who owns that query.

## 7. Image-library response flow

`GET /ingestion/images` loads the authenticated user's Postgres image rows. For
each stored image, the backend may exchange its stable Supabase object key for
a temporary signed URL. The API returns image metadata, state, face count, and
the temporary URL; it never exposes the Supabase service-role key.

There is also a separate browser-only photo path in `frontend/src/lib/photo-store.ts`.
Those photos are converted to data URLs and retained in `localStorage` (capped
at 24). They do not automatically enter Postgres, Supabase, InsightFace, or
Qdrant and therefore are not searchable through the backend unless submitted
through the search API.

## 8. Persistence map

| Data | Location | Lifetime/notes |
| --- | --- | --- |
| Google profile and Drive tokens | `users` in Postgres | Updated at Google session creation |
| Backend JWTs | Browser `localStorage` | Access JWT expires; refresh JWT renews access |
| Folder selection and progress | `user_folders` in Postgres | Durable |
| Ingestion task message | Redis/Celery | Carries the job UUID for worker delivery |
| Job progress and failures | `ingestion_jobs` in Postgres | Durable |
| Drive metadata | `images` in Postgres | Unique per user and Drive file ID |
| Original Drive bytes | Google Drive | Source of truth |
| Optional copied image bytes | Supabase Storage | Stable private object path |
| Face box and relationships | `faces` in Postgres | Durable relational metadata |
| Face embedding | Qdrant | One point per face UUID |
| In-memory ingestion bytes | Worker process | Discarded after file processing |
| Query image bytes and embedding | API process memory | Discarded after search |
| Query filename and metrics | `search_queries` in Postgres | Durable search history |
| Ranked matches | `search_results` in Postgres | Durable; linked to images/faces |
| Result image URL | Generated signed URL | Temporary; regenerated on later API reads |
| Browser-only uploaded photos | Browser `localStorage` | Separate from backend ingestion/search |

## 9. Tables that are present but not yet an active flow

`person_clusters` and `clustering_jobs` exist in the schema, and faces can hold
a `cluster_id`, but the current routers and services do not expose or run a
complete clustering workflow. They should be treated as schema groundwork,
not as a currently functioning stage of ingestion or search.
